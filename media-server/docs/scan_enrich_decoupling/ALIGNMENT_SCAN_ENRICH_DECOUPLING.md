# ALIGNMENT: 扫描与丰富化解耦设计

## 1. 背景与现状
**现状**：
- `UnifiedTaskScheduler` 在 `_execute_combined_task` 方法中直接耦合了元数据任务的创建逻辑。
- 扫描任务完成后，直接根据扫描结果（`new_file_ids`）创建 `TaskType.METADATA_FETCH` 任务。
- 这种紧耦合导致扫描逻辑与业务逻辑（元数据丰富、通知等）绑定，难以独立扩展和测试。

**目标**：
- 实现扫描与丰富化的完全解耦。
- 扫描仅产出“发现事件”（如 `NewFilesDiscovered`）。
- 丰富化作为独立的消费者监听事件，自行决定如何处理（创建任务、过滤等）。

## 2. 核心需求
1.  **事件驱动**：引入轻量级事件总线（Event Bus）。
2.  **发布事件**：扫描完成后，`UnifiedTaskScheduler` 发布 `ScanCompleted` 或 `NewFilesDiscovered` 事件。
3.  **消费事件**：独立的 `MetadataEnrichmentHandler` 监听事件，负责创建元数据丰富任务。
4.  **数据结构解耦**：事件载荷（Payload）应包含必要的最简信息（Storage ID, File IDs, etc.），不传输庞大的对象。

## 3. 边界确认
- **包含**：
    - 定义事件模型。
    - 实现简单的进程内事件总线（考虑到目前架构可能是单体或简单微服务，暂不引入复杂MQ，除非必要，这里先假设进程内解耦）。
    - 改造 `UnifiedTaskScheduler` 移除直接的任务创建代码。
    - 实现 `EnrichmentEventHandler`。
- **不包含**：
    - 复杂的分布式消息队列（Kafka/RabbitMQ）的引入（除非项目已有）。
    - 改变元数据丰富本身的核心逻辑（只改变触发方式）。

## 4. 疑问与决策点
- **Q1**: 是否需要持久化事件？
    - *决策*: 暂时不需要。如果处理失败，依赖任务重试机制或重新扫描。扫描任务本身是幂等的。
- **Q2**: 事件总线的实现方式？
    - *决策*: 使用 Python 的 `blinker` 库或简单的 `Observer` 模式实现进程内通信。如果项目没有 `blinker`，实现一个简单的 `EventBus` 类。
- **Q3**: 现有组合任务（Combined Task）如何处理？
    - *决策*: 组合任务本质上是“扫描+自动丰富”。解耦后，组合任务的语义变为“扫描开启自动丰富开关”，扫描器在发布事件时可以带上“需要丰富”的标记，或者丰富化处理器默认对所有新文件进行丰富，通过配置控制。
    - *修正*: 组合任务作为一种TaskType，其流程控制应由调度器编排，还是完全拆分？
    - *方案*: 扫描任务参数中 `enable_metadata_enrichment` 仍然保留，作为事件的一个属性传递。消费者根据此属性决定是否执行丰富化。

## 5. 验收标准
1.  `UnifiedTaskScheduler` 中不再包含 `TaskType.METADATA_FETCH` 的创建代码。
2.  扫描任务执行成功后，能够正确触发元数据丰富任务。
3.  可以通过配置或参数关闭丰富化，此时扫描后不触发丰富任务。
