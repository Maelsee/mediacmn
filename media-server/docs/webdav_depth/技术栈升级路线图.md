# 技术栈升级路线图（FastAPI 后端）

## 目标与范围
- 提升稳定性、吞吐与可观测性；降低维护成本；支持异构存储与事件驱动架构。
- 不破坏现有 API；灰度迁移，支持快速回滚。

## 总览
- 分四阶段执行，每阶段可独立验收与回滚。
- 每项包含产出物、迁移步骤、风险与回滚策略、验收标准。

---

## 阶段A：基础设施与质量栈升级（2–3周）
- 产出物
  - `uvicorn[standard] + uvloop + httptools` 启用；`orjson` 全局序列化；`pydantic v2` 升级；`Ruff + MyPy + Pyright` 引入；`Alembic` 迁移规范化。
- 步骤
  - 升级运行时：`uvicorn` 启用 `uvloop/httptools`；配置 `orjson` 为 FastAPI JSON 响应编码器。
  - 类型与校验：迁移 `pydantic v2`，修正 `BaseModel` 使用（`model_validate`, `model_dump`）。
  - 质量工具：加入 `ruff/mypy/pyright`，设置 CI `pre-commit` 钩子与最低通过门槛。
  - 数据迁移：将手写迁移整理入 `Alembic`，验证 `upgrade/downgrade` 可用。
- 风险与回滚
  - `pydantic v2` API 差异导致类型错误；保留 v1 分支与兼容适配器，必要时回滚。
- 验收
  - 所有接口通过回归；类型检查零高危；`Alembic` 成功执行并可回滚。

---

## 阶段B：数据层与连接池优化（2–3周）
- 产出物
  - `SQLAlchemy 2.x` 原生 async；Postgres 连接池切换 `asyncpg`；慢查询日志与指标。
- 步骤
  - ORM 升级：将核心会话迁移到 `SQLAlchemy 2.x` async；保留 `SQLModel` 作为薄层或分步替换为 `SQLAlchemy + Pydantic`。
  - 连接池：使用 `asyncpg`；开启 statement cache；连接池容量与超时参数化。
  - 指标：添加 DB 延迟、池占用率、超时次数的 Prometheus 指标。
- 风险与回滚
  - 部分查询生成差异；保留原会话封装，支持切换后端驱动。
- 验收
  - 主要接口 P95 延迟下降 ≥20%；慢查询可定位；无连接泄露与超时报警。

---

## 阶段C：队列与任务执行器替换（3–4周）
- 产出物
  - 队列层替换为 `Arq`（轻量）或 `Celery + Redis/RabbitMQ`（完整）；加入死信、延迟重试与监控面板。
- 步骤
  - 选型：若优先轻量与纯 async，选 `Arq`；若需求完整路由/调度/定时任务，选 `Celery`。
  - 适配：将现有 `TaskQueueService` 封装为适配层；把“组合任务”改为事件发布，由消费者自行创建元数据任务（参考架构文档）。
  - 监控：队列深度、吞吐、失败率与重试次数暴露到 Prometheus；必要时 Grafana 面板。
  - 灰度：扫描执行器先切到新队列，元数据执行器跟随；保留旧实现并行运行一周。
- 风险与回滚
  - 新队列配置不当引起堆积；保留旧队列切换开关；设置最大重试与死信回收。
- 验收
  - 队列卡点消除；失败重试与死信路径生效；无任务丢失。

---

## 阶段D：可观测性与稳定性增强（2周并行开展）
- 产出物
  - `structlog/loguru` JSON 日志；`OpenTelemetry` tracing；`Prometheus + Grafana` 指标面板；断路器与限流统一。
- 步骤
  - 日志：将服务与任务执行器统一输出结构化日志，注入 `traceId/taskId/storageId/fileId`。
  - 指标：API 延迟、扫描吞吐、刮削成功率、限流触发、断路器状态、Redis/Kafka 指标完整暴露。
  - Tracing：在“扫描→事件→丰富化→持久化→侧车”链路加入 trace 与 span；跨服务传播上下文。
  - 保护：采用 `aiolimiter` 令牌桶限流；`pybreaker` 断路器包装第三方插件调用。
- 风险与回滚
  - Tracing 与日志量增加带来开销；设置采样与切分到低成本存储（ELK/ClickHouse）。
- 验收
  - 面板可观测、报警可用；熔断与限流工作正常；线上稳定无雪崩。

---

## 阶段E：存储与扫描源抽象统一（2–3周）
- 产出物
  - 引入 `ScanSource` 适配层；Cloud 存储（如 S3/GDrive）枚举适配器上线；路径归一策略落地。
- 步骤
  - 在扫描引擎中实现 `ScanSource`：统一消费 `StorageClient`，产出 `StorageEntry` 流；处理分页、递归与重试。
  - 完成 Cloud 适配器：`aioboto3`（S3）或 `pydrive2`（GDrive），只读枚举先行；统一路径规范与相对路径持久化。
  - 压测：大目录枚举 P95 目标与 429/5xx 重试策略测试。
- 风险与回滚
  - Cloud API 限流与鉴权失败；保留旧实现与适配器开关。
- 验收
  - 新增任意存储类型只需注册工厂，不改扫描引擎；枚举吞吐稳定。

---

## 阶段F：多类型丰富化与持久化策略化（2–3周）
- 产出物
  - 丰富化插件能力声明与类型扩展；持久化策略注册中心；`sub_kind` 增量迁移。
- 步骤
  - 类型扩展：在 `MediaType` 增加综艺/动画/纪录片等；解析器映射“期/话/特别篇”，填充 `episode_type/absolute_episode_number`。
  - 策略化持久化：实现 `PersistenceStrategy` 接口与注册中心；保留现有 `_apply_*` 方法作为默认策略；新增类型按策略映射。
  - 数据库增量：为 `MediaCore` 增加 `sub_kind`；启用 `EpisodeExt.absolute_episode_number` 并建索引（见评估文档）。
- 风险与回滚
  - 旧数据与新字段不一致；迁移脚本设置默认 `NULL` 并逐步填充。
- 验收
  - 新类型端到端打通；老类型不受影响；事务幂等与回退有效。

## 工具与依赖清单
- 运行时：`uvicorn[standard]`、`uvloop`、`httptools`、`orjson`。
- ORM/DB：`SQLAlchemy 2.x`、`asyncpg`、`Alembic`。
- 队列：`Arq` 或 `Celery + Redis/RabbitMQ`。
- 观察：`structlog/loguru`、`Prometheus + Grafana`、`OpenTelemetry`。
- 保护：`aiolimiter`、`pybreaker`。
- 质量：`ruff`、`mypy`、`pyright`、`pre-commit`。
- 存储：`aioboto3`、`pydrive2`、`aiofiles`。

## 验收度量（示例目标）
- API P95 响应时间下降 ≥30%；扫描吞吐提升 ≥50%。
- 刮削失败率降低至 ≤2%；第三方限流触发降低 ≥80%。
- 队列任务丢失率 0；死信自动回收覆盖 100%。
- 指标覆盖率 90%+；关键链路 tracing 可视化。

## 风险控制与回滚策略
- 每阶段灰度发布与并行运行；保留旧实现开关与配置信号。
- 设定 SLIs/SLOs 与报警阈值；不达标立即回滚并记录根因。
