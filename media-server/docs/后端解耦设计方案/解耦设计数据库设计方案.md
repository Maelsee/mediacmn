# 媒体扫描-刮削-持久化解耦与数据库重构设计方案

## 1. 目标与范围
- 解耦后端 FastAPI 媒体处理流水线的三个阶段：扫描、刮削（丰富化）、持久化，确保模块独立、可扩展与可替换。
- 以 Redis 为任务中间件，形成生产者-消费者流水线：扫描→刮削→持久化，对应 3 个独立队列。
- 存储服务抽象为适配器（WebDAV/SMB/Local/Cloud），统一文件枚举、属性探测与数据访问端口。
- 刮削与持久化解耦，支持批量持久化以降低数据库写入开销。
- 数据库模型重构以支持更多类型（电影、剧集、综艺、动画、纪录片、特别篇等），提供规范化主源与类型差异适配。
- 结合 `DEVELOPMENT.md` 中 TMDB 元数据（参考 2833-3915 行片段），定义插件结果接口与刮削流程。

## 2. 设计原则
- 模块内聚、模块间解耦：扫描只产出“候选媒体文件”与基础探测结果，不依赖持久化；刮削专注元数据归一化；持久化专注批量写入与幂等。
- 接口优先：明确各阶段输入/输出契约，采用类型安全的消息结构与幂等键，便于横向扩展与回放。
- 幂等与去重：通过外部源唯一键、文件指纹与组合唯一约束，保证重复任务不产生脏写。
- 可观测与容错：队列采用消费组、重试与死信队列；每阶段记录结构化日志与指标。
- 最小耦合：存储适配器不含业务逻辑；插件不直接写库；持久化不依赖外部协议。

## 3. 技术栈推荐
- Web 框架：FastAPI（异步、类型安全、OpenAPI 输出与依赖注入）
- 配置与数据模型：Pydantic v2（消息体建模与验证）
- 任务中间件：Redis Streams + Consumer Groups（高吞吐可回放）
- Redis 客户端：`redis-py` 或 `redis.asyncio`（生产环境优先异步）
- HTTP 客户端：`httpx`（插件拉取远端元数据）
- 数据库：PostgreSQL + SQLAlchemy + Alembic（迁移与批量写）
- 存储适配：
  - WebDAV：`webdavclient3` 或基于 `httpx` 的轻量实现
  - SMB：`pysmb`
  - 本地：`pathlib`/`os`
  - Cloud：S3（`aioboto3`）或兼容的云存储 SDK（按需）
- 任务执行：独立 Worker 进程（`uvicorn` 提供 API；`python start_task_executor.py` 提供队列消费者）
- 可观测性：结构化日志（`loguru`/`structlog`）、指标（Prometheus + Grafana）

## 5. 接口设计（契约）

### 5.1 扫描引擎接口（ScanSource）
- 目标：定义扫描阶段的抽象，不依赖具体存储实现。
- 角色：生产者，负责将候选媒体文件推送至 `scan:stream`。
- 核心方法（语义描述）：
  - `enumerate(paths, filters) -> AsyncIterator<FileCandidate>`：遍历路径并产出候选文件。
  - `probe(file) -> MediaProbe`：对文件进行基础探测（大小、时长、编码、分辨率、字幕/封面伴随文件等）。
  - `to_message(candidate, probe) -> ScanMessage`：序列化为队列消息。
- FileCandidate 字段：
  - `tenant_id`（多用户/隔离）、`storage_id`、`full_path`、`relative_path`、`filename`、`size`、`ctime`、`mtime`、`etag/hash`
  - `asset_role`（`video|audio|subtitle|nfo|image|other`）
- MediaProbe 字段：
  - `duration`、`container`、`video_codec`、`audio_codec`、`resolution`、`bitrate_kbps`、`audio_channels`、`hdr`、`exists_sidecar`（如 `.nfo`、`.srt`）
- ScanMessage（push 到 `scan:stream`）：
  - `idempotency_key`：`<tenant_id>:<storage_id>:<full_path>:<etag>`
  - `candidate`：见上
  - `probe`：见上
  - `media_type_hint`：`movie|series|episode|season|unknown`（基于路径启发，非强制）
  - `priority`：`normal|high`

### 5.2 存储服务接口（Storage Adapter）
- 目标：将外部协议（WebDAV/SMB/Local/Cloud）转换为统一的标准端口数据。
- 适配器通用能力：
  - `stat(path) -> FileStat`
  - `listdir(path, recursive=false) -> Iterator<FileStat>`
  - `open_stream(path, range?) -> BinaryStream`
  - `resolve_select_paths(select_path[]) -> Iterator<FileStat>`（支持按配置选择扫描子路径）
  - `compute_etag(path) -> str`（协议可用时）
- FileStat 字段：`full_path`、`relative_path`、`filename`、`size`、`mtime`、`is_dir`、`exists`
- 适配器实现示例：WebDAV（基于 `PROPFIND`/`GET`）、SMB（基于 `pysmb` 目录枚举）、Local（`pathlib`）、Cloud（S3 `ListObjectsV2`）。
- 配置约束：每个存储记录采用统一配置结构（参考 `DEVELOPMENT.md` 14-32 行示例的 `storage_type` 与 `config` 结构），由 FastAPI 管理与校验。

### 5.3 刮削（丰富化）接口与插件接口
- 刮削执行器（Metadata Enricher）：消费者与生产者，消费 `scan:stream`，产出 `enrich:stream`。
- 插件接口（Scraper Plugin）：负责对接 TMDB/TVDB/IMDB/豆瓣等外部源，并返回规范化结果。
- Plugin 合同：
  - `supports(kind, subtype) -> bool`：是否支持指定类型（电影/剧集/综艺/动画等）。
  - `identify(candidate, probe) -> ContentHint`：基于文件名、路径与探测结果识别候选（系列名、季/集号、年份等）。
  - `fetch_movie(tmdb_id|query) -> MoviePayload`
  - `fetch_series(series_id|query) -> SeriesPayload`
  - `fetch_season(series_id, season_number) -> SeasonPayload`
  - `fetch_episode(series_id, season_number, episode_number|absolute_number) -> EpisodePayload`
- ContentHint 字段：`title`、`year`、`season_number?`、`episode_number?`、`absolute_episode_number?`、`kind`、`subtype`（`Scripted|Reality|Animation|Documentary|Special|Other`）。
- 规范化结果（NormalizedMetadataGraph）：
  - `core`：对应 `content_core`（见数据库重构）
  - `external_ids[]`：源与键（如 `tmdb_id`、`imdb_id`）
  - `detail`：`movie_detail|series_detail|season_detail|episode_detail` 之一
  - `relations[]`：系列-季-集关系骨架
  - `artwork[]`：`poster|backdrop|still|fanart...`
  - `credits[]`：`cast|crew|guest`（综艺嘉宾为 `guest`）
  - `raw`：保留原始 JSON（jsonb）
- 类型差异适配（结合 TMDB 返回示例，参考 `DEVELOPMENT.md` 2833-3315）：
  - 综艺：`series_detail.series_type = Reality`；`status = Returning Series`；`in_production = true`；`episode_run_time` 通常较长；嘉宾入 `credits.role = guest`。
  - 动画：优先 `absolute_episode_number` 支持；允许无季模型（直接 `series_has_episode`）。
  - 特别篇：`episode_type = special`；可能挂在 `season_number = 0` 或直接挂系列。

### 5.4 持久化服务接口（Metadata Persistence Service）
- 目标：与刮削完全解耦，批量持久化 `NormalizedMetadataGraph` 与文件资产映射，降低数据库操作次数。
- 输入：`PersistBatchMessage{ graphs[], file_assets[] }`（从 `enrich:stream` 聚合或单条透传）
- 行为：
  - 对 `content_core` 执行 `bulk upsert`（幂等键：`(user_id, canonical_source, canonical_external_key)` 或 `canonical_tmdb_id`）
  - 写入/更新 `detail` 表（`movie_detail|series_detail|season_detail|episode_detail`）
  - 建立 `content_relation` 骨架（系列-季-集）
  - 写入 `external_id`、`artwork(type=poster/backdrop/still)`、`credit`
  - 映射 `file_asset` 与 `media_version`（通过 `asset_role` 与 `variant_fingerprint`）
- 输出：`PersistResult{ inserted, updated, skipped, errors[] }`
- 事务与批量：每批采用事务边界，遇错误回滚并分片重试；按类型与源分桶，控制单批大小（如 100-500 条）。

## 6. Redis 队列与任务流程

### 6.1 队列命名与拓扑
- `scan:stream`：ScanSource 生产；Enrich Workers 消费。
- `enrich:stream`：Enrich Workers 生产；Persist Workers 消费。
- `persist:stream`：Persist Workers 消费（可作为审计或对下游通知）。
- `deadletter`：各阶段失败消息进入死信流，保留错误与可重放信息。

### 6.2 消费组与并发
- 每个 Stream 建立一个或多个 Consumer Group（如 `scan-cg`、`enrich-cg`、`persist-cg`），每组下多实例并发消费。
- 使用 `XADD`/`XREADGROUP`/`XACK`；超时未确认的挂起项采用 `XPENDING`+`XCLAIM` 进行再分配。
- 消息体包含 `idempotency_key`，消费端先查重（Redis Set/DB 唯一键），确保幂等。

### 6.3 消息结构（统一 Envelope）
- 通用字段：
  - `tenant_id`、`task_id`、`stage`（`scan|enrich|persist`）、`timestamp`、`retry_count`、`idempotency_key`
- 负载字段：
  - `scan_payload`：`FileCandidate` + `MediaProbe` + `media_type_hint`
  - `enrich_payload`：`NormalizedMetadataGraph` + `source_provider`（`tmdb|tvdb|douban|imdb`）
  - `persist_payload`：`PersistBatchMessage`
- 错误字段：`error_code`、`error_message`、`trace_id`、`previous_stage_payload_ref`

### 6.4 重试与死信策略
- 阶段内重试采用指数退避（如 3 次，`0.5s/2s/10s`）。
- 超过上限将消息写入 `deadletter`，包含错误原因与重放指针（原 Stream ID）。
- 管控面提供重放 API（从 `deadletter` 将选定消息重推至对应 Stream）。

## 7. 功能模块间交互逻辑
- 扫描阶段：
  - FastAPI 配置存储源与扫描计划（选择路径、忽略规则、并发度）。
  - Scan Workers 通过 Storage Adapter 枚举文件，生成 `FileCandidate` 与 `MediaProbe`，写入 `scan:stream`。
- 刮削阶段：
  - Enrich Workers 消费 `scan:stream`，基于 `ContentHint` 调用插件（如 TMDB 的 `/movie/{id}`、`/tv/{id}`、`/tv/{id}/season/{n}`、`/episode`，参考 `DEVELOPMENT.md` 示例）。
  - 插件返回 `NormalizedMetadataGraph`，包含系列/季/集层级、综艺/动画差异字段与 `artwork(still)` 等。
  - Enrich Workers 将规范化图打包为 `PersistBatchMessage`（可按系列/季聚合），写入 `enrich:stream`。
- 持久化阶段：
  - Persist Workers 消费 `enrich:stream`，对照数据库幂等键进行 `bulk upsert`；建立层级关系与外部 ID 链接；落地 `file_asset`。
  - 成功后可写入 `persist:stream` 供审计或通知（如触发转码或封面下载）。

## 8. 数据库重构要点（与实现对齐）
- 核心表 `content_core`：承载通用信息与规范化主源（`canonical_source`/`canonical_external_key`/`canonical_tmdb_id`），唯一约束确保幂等。
- 层级关系 `content_relation`：统一表示系列-季-集（含 `series_has_episode` 直挂集），支持无季剧与绝对集序（动画）。
- 类型扩展表：`movie_detail`、`series_detail`、`season_detail`、`episode_detail` 覆盖类型差异；综艺采用 `series_detail.series_type=Reality`，动画采用 `episode_detail.absolute_episode_number`。
- 通用关联表：`external_id`、`genre`/`content_genre_link`、`person`/`credit`、`artwork(type 含 still)`、`media_version`、`file_asset`、`collection`/`content_collection_link`。
- 查询路径优化：电影按 `canonical_tmdb_id` 或 `external_id` 查；剧集按 系列-季-集 唯一键；综艺按 `in_production` 与最近 `air_date`；动画按 `absolute_episode_number`。
- 迁移与落地：采用 Alembic 建表与迁移；逐步迁移旧数据至新结构并替换读路径。

（详细字段、唯一键与索引请参考同目录《数据库重构.md》，本方案与其一致并在类型差异与队列契约上进行补充。）

   - 在 `services/media/metadata_persistence_service.py` 实现 Persist Worker（批量 upsert 与事务）。
2) 队列与执行器：
   - 配置 Redis（参考 `docker-compose.yml` 中 `redis` 服务，端口 `9001` 与密码）。
   - 建立 Streams 与 Consumer Groups；实现生产/消费、ACK、重试、死信。
   - 为 `start_task_executor.py` 引入并行消费者（Scan/Enrich/Persist 三类）。
3) 数据库与迁移：
   - 依据《数据库重构.md》创建/更新表与索引（Alembic）。

4) 插件落地与类型适配：
   - TMDB 插件：实现 `/movie`、`/tv`、`/season`、`/episode`，映射综艺与动画差异字段。

5) 批量持久化优化：
   - 聚合策略（按系列/季/文件批次）；控制批量大小与事务边界。

6) 可观测与管控：
   - 结构化日志与指标；Pending/Deadletter 可视化。
   - 管控 API：重放、暂停/恢复、并发度与优先级调整。

## 10. 验收标准
- 三阶段独立进程与接口契约完成；队列消息结构稳定并记录在文档中。
- 综艺与动画类型完整支持（按示例字段入库），可正确查得最近播出与绝对集序。
- 刮削与持久化解耦，持久化支持批量 upsert，性能优于逐条写入。
- 幂等与去重有效：重复扫描/刮削不会产生重复记录或脏数据。
- 观察性与容错：死信与重试链路可用，具备重放能力。

## 11. 风险与缓解
- 外部源限制（速率/不可用）：引入速率限制与缓存；失败进入死信并延迟重试。
- 文件命名不规则：增强 `identify` 规则与人工校正接口；保留原始 `raw_data` 便于人工复核。
- 大批量写入事务冲突：分桶与分片；降低单事务体量并重试冲突片段。

## 12. 接口契约（结合当前项目）
- 扫描引擎契约（新增抽象）：

- 任务与队列：
  - 现有任务队列服务在 `services/task/task_queue_service.py:101`（Redis ZSet 优先级队列）；统一调度在 `services/task/unified_task_scheduler.py:69`；扫描引擎在 `services/scan/unified_scan_engine.py:459`。
  - 本方案引入 Redis Streams（`scan:stream`、`enrich:stream`、`persist:stream`），用作阶段间消息流；可与现有 TaskQueue 并存，逐步迁移。

## 13. Redis 消息契约
- Stream 名称：`scan:stream`、`enrich:stream`、`persist:stream`、`deadletter`
- 消费组：`scan-cg`、`enrich-cg`、`persist-cg`
- Envelope（字段）：
  - 基础：`tenant_id`、`task_id?`、`stage`、`timestamp`、`retry_count`、`idempotency_key`
  - 载荷：
    - `scan_payload`：`candidate`（JSON）、`probe`（JSON）、`media_type_hint`
    - `enrich_payload`：`normalized_graph`（JSON）、`source_provider`
    - `persist_payload`：`batch`（JSON）
  - 错误：`error_code`、`error_message`、`trace_id`、`previous_stage_payload_ref`
  - 示例推送：`XADD scan:stream * {tenant_id, storage_id, idempotency_key, candidate, probe, media_type_hint, priority}`
