# 扫描与存储解耦架构设计方案

## 1. 设计目标
- **解耦原则**：遵循 Ports & Adapters (Hexagonal) 架构。
  - **Ports (端口)**：扫描引擎定义需要的抽象接口（`ScanSource`），不依赖具体存储库。
  - **Adapters (适配器)**：存储服务提供具体实现（WebDAV/SMB/Local/Cloud 适配器），将外部协议转换为标准端口数据。
- **透明性**：扫描核心逻辑（分类、入库、处理）完全不感知底层是本地文件还是 S3 对象。
- **弹性与流式**：支持大规模文件流式枚举，内置背压控制、断点续传与故障容错。

## 2. 核心抽象层设计

### 2.1 统一存储契约 (Existing & Enhanced)

### 2.2 适配器层设计 (Adapters)
位于 `services/storage/adapters/`。负责将 `StorageClient` 的输出转化为 `ScanEntry`。

#### `StorageClientAdapter`
通用适配器，包装任何符合 `StorageClient` 接口的实例。

- **职责**：
  - 调用 `client.list_dir` (递归或非递归)。
  - 路径归一化：将协议特有的绝对路径（如 `smb://server/share/movie/a.mp4`）转化为统一相对路径（`/movie/a.mp4`）。

## 3. 扫描引擎重构 (UnifiedScanEngine)

### 3.1 扫描管线 (Pipeline)
扫描引擎不再直接调用 `StorageService`，而是接收一个 `ScanSource` 实例。

**流程：**
1. **Source Stage**: `ScanSource.enumerate_entries()` 产出 `ScanEntry` 流。
2. **Filter Stage**: 过滤忽略的文件（如 `.DS_Store`）和非媒体类型（基于扩展名/MIME）。
3. **Batch Stage**: 将流聚合为 Batch (e.g., 100 items) 以减少数据库 IO。
4. **Process Stage**: 并行处理 Batch。
   - `FileAssetProcessor`: 幂等 Upsert `FileAsset` 表。
   - `DeleteSyncProcessor`: 标记本次扫描未遇到的文件（软删除）。

### 3.2 依赖注入
在 `UnifiedScanEngine.scan_storage` 方法中，不传入 `storage_id` 让引擎去查库，而是传入构造好的 `ScanSource`。

```python
# 伪代码：重构后的扫描入口
class UnifiedScanEngine:
    async def scan(self, source: ScanSource, processor: ScanProcessor, options: ScanOptions):
        async for batch in batch_stream(source.enumerate_entries(), size=100):
            await processor.process_batch(batch)
```

这样，引擎完全不知道 `storage_id` 的存在，只处理数据流。`storage_id` 仅在构造 `ScanSource` 时由调度层使用。

## 5. 关键技术点与策略

### 5.1 路径归一化 (Path Normalization)
- **问题**：WebDAV 返回 `/dav/movies/avatar.mkv`，SMB 返回 `\movies\avatar.mkv`。
- **策略**：
  - **存储根 (Root)**：配置中定义的挂载点（如 WebDAV 的 `root_path`）。
  - **相对路径 (Relative Path)**：所有业务逻辑（刮削、播放、字幕）均使用 **相对于存储根的路径**。
  - **适配器职责**：`StorageClientAdapter` 必须剥离协议前缀和 Root 前缀，确保 `ScanEntry.path` 始终以 `/` 开头且不含协议信息。

### 5.2 分页与流式 (Pagination & Streaming)
- 对于 Cloud (S3) 和 WebDAV，目录可能非常大。
- **StorageClient 增强**：建议在 `list_dir` 中增加 `pagination_token` 支持（未来优化）。
- **适配器实现**：
  - 当前：适配器内部处理分页，对外暴露 `AsyncIterator`。
  - 优势：扫描引擎不需要知道 S3 需要 `ContinuationToken`，只需 `async for item in source`。

### 5.3 错误隔离 (Fault Isolation)
- **场景**：SMB 链接在扫描中途断开。
- **策略**：
  - `ScanSource` 内部捕获连接异常。
  - 实现 **指数退避重试 (Exponential Backoff)**。
  - 若重试耗尽，抛出 `ScanSourceError`。
  - 引擎捕获该错误，记录日志，标记该 Batch 失败，但不一定终止整个扫描（取决于配置 `fail_fast`）。

## 6. 实施步骤

1. **定义接口**：在 `services/scan/interfaces.py` 中定义 `ScanSource` 和 `ScanEntry`。
2. **实现适配器**：在 `services/storage/adapters/` 实现 `StorageClientAdapter`，对接现有的 `StorageClient`。
3. **重构引擎**：
   - 修改 `UnifiedScanEngine`，使其接收 `ScanSource` 而非 `storage_id`。
   - 将原有的 `_recursive_scan` 逻辑移入 `StorageClientAdapter` 或通用的 `RecursiveEnumerator`。
4. **调度层适配**：
   - 修改 `UnifiedTaskScheduler`，在创建任务时，先通过 `StorageService` 获取 `Client`，封装为 `Adapter`，再传给引擎。
5. **测试验证**：
   - 单元测试：Mock `ScanSource`，验证引擎入库逻辑。
   - 集成测试：使用 Local 存储验证路径归一化和递归逻辑。
